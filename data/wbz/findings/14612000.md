## Notes

- Geometries served via Dresden's KommISDD OGC API Features endpoint (cardogis WFS3 server). All collections also accessible via classical WFS; both shp and gpkg derivable client-side from the GeoJSON.
- BTW 2021 layer L703 schema: objectid (Int64), wb (Wahlbezirk-ID, Text), wb_text (Text label), shape (Polygon). BTW 2025 layer L1650 has analogous structure.
- Other electoral geometry layers in catalog (for context, not requested): L702 / L1651 (Wahlkreise BTW 2021/2025), L1091 / L1588 (Landtagswahl 2019/2024 Wahlbezirke), L1542 / L1565 (Kommunalwahl 2019/2024), L1063 (OB-Wahl 2022), L284 / L1558 (Kommunalwahl Wahlkreise), L285 / L1559 (Landtag Wahlkreise).
- BTW 2013 and 2017: no published geometry layer found on opendata.dresden.de, KommISDD catalog, GovData, or geoportal.sachsen.de. Sachsen state geoportal hosts only Verwaltungsgrenzen (Kreis/Gemeinde), not Wahlbezirke. Earliest available federal-election Wahlbezirk geometry is 2021.
- For 2013/2017 historical boundaries, primary path is a Verwaltungsanfrage to wahlamt@dresden.de cc geoservice@dresden.de; secondary path FragDenStaat (existing Dresden requests visible). Wahlbezirk-Einteilungen for 2013/2017 likely exist internally as ALKIS/KommISDD historical layers but were not published as open data.
- Feature counts verified live via curl on 2026-05-04: BTW 2021 = 365 polygons, BTW 2025 = 399 polygons. Counts match "~400 WBZ" expectation and exclude the 214 Briefwahlbezirke (no geometry).
- offenesdresden/wahlbezirke (https://github.com/offenesdresden/wahlbezirke) demonstrates the BTW 2021 endpoint usage (fetch.sh) — useful as a reproducible code example.
